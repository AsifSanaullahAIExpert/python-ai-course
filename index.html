How to follow this course

The approach that produces real, lasting results

Type every example — never copy-paste

The most important rule: type every code example by hand. Copying and pasting produces the illusion of learning. When you type, your brain processes every character. Every typo you make and fix is teaching you something no amount of reading can.

Your learning loop on each page

  1. Read the explanation carefully before touching the keyboard
  2. Type the code example — every single character
  3. Run it and observe the output
  4. Change one value or condition — see what happens
  5. Break it deliberately and read the error message carefully
  6. Fix the error yourself before looking ahead
  7. Move on only when you can explain what the code does

When you get stuck

  1. Read the error message carefully — Python errors are precise and informative
  2. Re-read the lesson for the relevant concept
  3. Search the exact error text in a web browser
  4. Ask an AI assistant to explain the error — not write the fix for you

Ask AI to explain errors, not solve them. "What does this error mean and why does it happen?" is far more valuable than "Fix my code." Understanding is the goal — not a working program you cannot modify.

Understanding Python

What Python is and how it executes your code

A high-level, interpreted language

Python is a programming language — a formal system for giving instructions to a computer. "High-level" means it manages memory and processor details automatically so you can focus on what you want to accomplish, not how the hardware executes it.

When you run a Python file, an application called the Python interpreter reads it top to bottom, one line at a time, executing each instruction immediately. No compilation step is needed — write a line, run it, see the result instantly.

# Python executes each line in sequence, immediately
print("Loading AI model...")
model = "gpt-4o-mini"
print(f"Ready: {model}")
# Output:
# Loading AI model...
# Ready: gpt-4o-mini

Dynamically typed

Python determines a variable's type automatically from the value you assign. You never declare types manually (though optional type hints exist for clarity).

# Python infers the type from the assigned value
model_id    = "gpt-4o"     # str — text
max_tokens  = 1024          # int — whole number
temperature = 0.7           # float — decimal
streaming   = True          # bool — True or False

Always use Python 3.11 or newer. Python 2 was retired in 2020 and is incompatible with all modern AI libraries. Every reference in this course assumes Python 3.

Organising your project

Folder structure that scales from day one

Open a folder, not a file

VS Code works best when you open an entire project folder, not individual files. This gives you the file tree, an integrated terminal rooted in your project, and Git tracking from the start.

A clean starter structure

my-ai-project/
├── main.py           # entry point — where execution starts
├── utils.py          # reusable helper functions
├── config.py         # constants and configuration values
├── .env              # secret keys — NEVER commit this file
├── .env.example      # template with empty values — safe to commit
├── .gitignore        # tells Git what to ignore
├── requirements.txt  # or pyproject.toml when using uv
└── README.md         # explains what the project does and how to run it

Open a folder in VS Code

  1. Create a new folder: e.g. my-ai-project
  2. Open VS Code
  3. Go to File → Open Folder and select it

Integrated terminal shortcut

Press Ctrl+` (backtick) to open a terminal at the project root. This is where you run Python and pip commands.

One folder per project, always opened at the root. This single discipline eliminates almost all path confusion that trips beginners up.

Virtual environments

Isolate every project's dependencies

Why isolation matters

Suppose you have two projects: one needs OpenAI SDK v1.3.0, another requires v1.61.0. Without isolation, installing the newer version breaks the first project. Virtual environments give each project its own private Python and package set.

Create and activate

python -m venv .venv
.venv\Scripts\activate
# Prompt becomes (.venv)
python3 -m venv .venv
source .venv/bin/activate
# Prompt becomes (.venv)

Complete workflow

# 1. Create — once per project
python -m venv .venv

# 2. Activate — every new terminal session
source .venv/bin/activate

# 3. Install packages for this project
pip install openai anthropic pandas

# 4. Deactivate when finished
deactivate

Always add .venv to your .gitignore file. Virtual environments are large and machine-specific — never commit them to Git. Commit requirements.txt instead.

What is programming?

Instructions, data, and the logic that connects them

Programming is precise instruction-giving

A program is a list of instructions a computer follows exactly, in order, without judgment or common sense. Your role is to express your intentions precisely enough that a machine can execute them without ambiguity.

# A program is a sequence of precise instructions
dataset_path = "/data/training.csv"
batch_size   = 32
print(f"Loading {dataset_path} in batches of {batch_size}")

Every program does three things

Input
Receive data from a file, API, keyboard, database, or sensor
Process
Transform that data — calculate, classify, filter, or generate
Output
Do something with the result — display, save, send, or trigger

Decomposition is the core skill

Great programmers break large vague problems into small precise instructions. "Build a recommendation engine" is not actionable. "If the user's last three queries were about machine learning, rank ML articles first" is. This skill — decomposition — is what distinguishes productive programmers from those who feel permanently stuck.

Programming is a skill, not a talent. The ability to think in precise logical steps is built through practice — not something you are born with.

Python syntax rules

The grammar Python enforces

Indentation defines code blocks

Python uses consistent indentation to define which lines belong inside a block. The standard is 4 spaces per level. Mixing tabs and spaces causes errors.

# ✅ Correct — 4 spaces, consistent
if temperature > 38.0:
    print("Fever detected")
    if temperature > 40.0:
        print("High fever — seek medical attention")

# ❌ Wrong — IndentationError at runtime
if temperature > 38.0:
  print("Fever")       # 2 spaces
    log("alert")        # 4 spaces — inconsistent!

Case sensitivity

Python is case-sensitive. model, Model, and MODEL are three distinct names. Convention: variables and functions use snake_case; class names use PascalCase; constants use UPPER_SNAKE_CASE.

Statements end at the line

No semicolons needed. Each line is one statement. For long expressions, wrap in parentheses to span multiple lines:

# Parentheses allow multi-line expressions
response = client.chat.completions.create(
    model="gpt-4o",
    messages=messages,
    max_tokens=512,
    temperature=0.7
)

Configure VS Code to use 4 spaces: Settings → Editor: Tab Size → 4, Editor: Detect Indentation → off, Editor: Insert Spaces → on.

Understanding error messages

Errors are precise — learn to read them, not fear them

Errors are information, not judgment

Error messages are Python explaining exactly what went wrong and where. Experienced developers read tracebacks the way doctors read X-rays — quickly and with purpose.

Anatomy of a traceback

Traceback (most recent call last):
  File "main.py", line 12, in <module>
    result = process_response(data)
  File "main.py", line 5, in process_response
    return data["choices"][0]["message"]
KeyError: 'choices'

Strategy: the last line names the error type and the exact problem. Lines above trace how execution arrived there. Start from the bottom and read upward.

The most common errors

SyntaxError
Python cannot parse the code — missing colon, bracket, or quote
NameError
You used a variable name that has not been defined yet
TypeError
Operation applied to the wrong type — adding string to integer
ValueError
Right type but invalid value — int("hello") fails
IndexError
Accessed list index that does not exist
KeyError
Accessed dictionary key that does not exist

Copy the exact error message into a web search. Someone has almost certainly hit the same error and documented the solution.

Code formatting

Write code that humans can read and maintain

Why formatting matters

Code is read far more often than written. Inconsistent formatting makes code hard to review, debug, and extend. PEP 8 is Python's official style guide. Learning its key rules early prevents significant friction later.

# ✅ Spaces around operators — easier to scan
token_limit = 1024
cost = tokens * 0.00015

# ❌ No spaces — harder to read
token_limit=1024
cost=tokens*0.00015

# ✅ Two blank lines between top-level functions
def load_data(path):
    pass


def process(data):
    pass

# ✅ Descriptive snake_case names
max_context_length = 128000
learning_rate = 0.001

Automate it with Ruff

Never format code manually. Install Ruff (covered in Essential Tools) and it reformats code automatically every time you save. Zero effort, consistent output on every file.

Good naming matters more than any other formatting rule. calculate_token_cost() needs no comment. calc() needs several. Write for the next person who reads your code.

Variables

Naming and storing values in memory

Variables are named storage

A variable is a name bound to a value. Writing model = "gpt-4o" stores the string and attaches the name model to it. Every time you write model afterward, Python retrieves that value.

# Assign with the = operator
model_id    = "gpt-4o-mini"
max_tokens  = 1024
temperature = 0.7
streaming   = True

# Use in expressions
print(f"Model: {model_id}, tokens: {max_tokens}")

# Reassign to a new value any time
max_tokens = 2048
print(max_tokens)   # 2048

Naming rules

  • Letters, digits, and underscores only — no spaces or hyphens
  • Cannot start with a digit: 2model invalid, model2 fine
  • Cannot use Python keywords: if, for, return, class, import
  • Variables and functions: snake_case — api_key, load_data
  • Constants: UPPER_SNAKE_CASE — MAX_RETRIES, BASE_URL

Multiple assignment

# Assign several variables in one line
host, port, db = "localhost", 5432, "ai_app"

# Swap two values without a temp variable
a, b = b, a

Name variables after what the value represents, not its type. user_email beats string1. Code that reads like English requires far fewer comments.

Comments

Writing explanations that help future readers — including yourself

Comment the why, not the what

Well-named code already tells you what it does. Comments should explain the reasoning — why a value was chosen, why this approach instead of another, what an edge case handles.

# ❌ Useless — just restates the code
count += 1  # add 1 to count

# ✅ Useful — explains the reasoning
retry_delay = 1.5  # exponential backoff: multiplied on each retry

# ✅ Useful — explains an edge case
if status_code == 429:
    # Rate limited — API requires a pause before retrying
    time.sleep(retry_delay)

Docstrings

Functions, classes, and modules should have docstrings — triple-quoted strings immediately after the definition that describe parameters and return values.

def call_ai(prompt, model="gpt-4o-mini", max_tokens=512):
    """Call the OpenAI API with a text prompt.

    Args:
        prompt: The question or instruction to send.
        model: Model identifier string.
        max_tokens: Upper limit on response length.

    Returns:
        The model response as a string.
    """
    pass  # implementation goes here

Introduction to functions

Write once, use everywhere — the DRY principle

Why functions matter

DRY stands for "Don't Repeat Yourself." Every time you copy and paste code instead of writing a function, you create a maintenance problem — when something changes, you have to update every copy. Functions solve this by giving a block of logic a name you can call from anywhere.

# Without functions — repeated code, hard to maintain
print("Welcome, Khalid!")
print("You have 3 new messages.")

print("Welcome, Mariam!")
print("You have 7 new messages.")

# With a function — write once, use everywhere
def greet_user(name, message_count):
    print(f"Welcome, {name}!")
    print(f"You have {message_count} new messages.")

greet_user("Khalid", 3)
greet_user("Mariam", 7)

The black box mental model

Think of a function as a black box: you put inputs in, something happens inside, and you get an output back. You do not need to know how it works — only what it takes and what it returns. This is why Python has thousands of useful functions you can use without reading their source code.

Scope

# Variables inside a function are local — invisible outside
def calculate_cost(tokens):
    rate = 0.00015       # local — only exists here
    return tokens * rate

print(calculate_cost(1024))   # 0.1536
# print(rate)  ← NameError: rate is not defined outside

A function should do one thing and do it well. If you find yourself writing "and" when describing what a function does, it probably should be two functions.

import requests, time

def safe_get(url, retries=3):
    for attempt in range(retries):
        r = requests.get(url)
        if r.status_code == 200:
            return r.json()
        if r.status_code == 429:  # rate limited
            time.sleep(2 ** attempt)  # exponential backoff
    raise RuntimeError(f"Failed after {retries} attempts")

Error handling overview

Why errors happen and how Python classifies them

The problem with unhandled errors

# Without error handling — crashes on any failure
import requests
data = requests.get(url).json()
print(data["choices"][0]["message"])
# ↑ Fails if: no internet, API down, key wrong, response format changed

Python's exception hierarchy

BaseException
├── Exception             # most errors inherit from this
│   ├── ValueError        # wrong value for the type
│   ├── TypeError         # wrong type for the operation
│   ├── KeyError          # dict key not found
│   ├── IndexError        # list index out of range
│   ├── AttributeError    # object has no such attribute
│   ├── FileNotFoundError # file does not exist
│   └── RuntimeError      # general runtime failure
└── KeyboardInterrupt     # Ctrl+C

Defensive programming principles

  • Validate inputs before processing them
  • Catch specific exceptions, not bare except:
  • Log errors with enough context to debug them later
  • Fail loudly (raise) rather than silently (pass)

Well-handled errors make your programs resilient. An AI app that handles API failures gracefully will work even when the network is flaky or the service is temporarily down.

else and finally

try:
    data = load_file("data.json")
except FileNotFoundError:
    print("File not found")
else:
    # runs only if no exception occurred
    process(data)
finally:
    # always runs — use for cleanup
    print("Done")

Custom exceptions

class TokenLimitError(ValueError):
    """Raised when prompt exceeds the token limit."""
    pass

def validate_prompt(prompt, limit=4096):
    if len(prompt.split()) > limit:
        raise TokenLimitError(f"Prompt exceeds {limit} tokens")

Never use bare except: without specifying an exception type. It catches KeyboardInterrupt and SystemExit, making your program impossible to stop cleanly.

Common errors and fixes

The six errors every Python developer encounters — and how to fix them

NameError — variable not defined

# ❌ Using before assigning
print(model_name)   # NameError: name 'model_name' is not defined

# ✅ Fix: define the variable first
model_name = "gpt-4o"
print(model_name)

TypeError — wrong type

# ❌ Adding string and integer
print("Cost: " + 0.002)   # TypeError

# ✅ Fix: convert to string first
print("Cost: " + str(0.002))
print(f"Cost: {0.002}")     # f-string handles conversion

KeyError — dictionary key missing

config = {"model": "gpt-4o"}
config["temperature"]   # KeyError: 'temperature'

# ✅ Fix: use .get() with a default value
config.get("temperature", 0.7)   # 0.7

IndexError — list index out of range

names = ["Khalid", "Rania"]
names[5]   # IndexError: list index out of range

# ✅ Fix: check length before indexing
if len(names) > 5:
    print(names[5])

AttributeError — object has no such attribute

text = "hello world"
text.append("!")   # AttributeError: str has no 'append'
# ✅ Fix: strings use + or f-strings for concatenation
text = text + "!"

ValueError — right type, wrong value

int("hello")   # ValueError: invalid literal for int()
# ✅ Fix: validate before converting
if user_input.isdigit():
    value = int(user_input)

Protecting secrets

Never commit API keys or passwords to Git

Never write API keys, passwords, or tokens directly in source code. GitHub scans public repositories for exposed keys and notifies providers — who may then revoke them immediately.

The safe pattern

# ❌ NEVER do this
client = OpenAI(api_key="sk-proj-abc123...")

# ✅ Do this instead
import os
from dotenv import load_dotenv

load_dotenv()
client = OpenAI(api_key=os.getenv("OPENAI_API_KEY"))

Your .gitignore must include these

# .gitignore
.env
.venv/
__pycache__/
*.pyc
.DS_Store

The .env.example pattern

Commit a file called .env.example with the key names but empty values. Anyone cloning your project knows what secrets to provide without seeing your actual keys.

# .env.example — safe to commit
OPENAI_API_KEY=
ANTHROPIC_API_KEY=
DATABASE_URL=

Environment variables

Read runtime configuration from the operating system

import os

# Read an environment variable
api_key = os.getenv("OPENAI_API_KEY")

# With a fallback default
model = os.getenv("AI_MODEL", "gpt-4o-mini")

# Fail fast if a required variable is missing
if not api_key:
    raise ValueError("OPENAI_API_KEY environment variable not set")

Setting variables in the terminal

# macOS / Linux — for this terminal session only
export OPENAI_API_KEY=sk-your-key-here

# Windows PowerShell — for this session only
$env:OPENAI_API_KEY = "sk-your-key-here"

In production

On servers and cloud platforms (AWS, Azure, Google Cloud), environment variables are set in the platform's dashboard or configuration files — never in your code. Your program reads them the same way: os.getenv().

Environment variables let the same code behave differently in development (using a test key) versus production (using the real key) — without any code changes.

Ruff formatter

The fastest Python linter and formatter — automatic code quality

What Ruff does

Ruff checks your code for style violations (linting) and automatically rewrites it to follow PEP 8 (formatting) — all in milliseconds. It replaces Black, isort, flake8, and pylint with a single tool that is 10–100× faster than any of them.

Install

pip install ruff

Use it

# Check for issues without fixing
ruff check main.py

# Format the file automatically
ruff format main.py

# Check and fix an entire project
ruff check . --fix
ruff format .

VS Code integration

  1. Install the Ruff extension from the VS Code marketplace
  2. Add to VS Code settings.json:
{
  "editor.formatOnSave": true,
  "editor.defaultFormatter": "charliermarsh.ruff"
}

Now every time you save a file, Ruff reformats it automatically. No manual formatting ever again.

Enabling format-on-save means you never think about formatting again. Just write code and save — Ruff handles the rest. Your code will always be consistent.

Modern Python features

Type hints, dataclasses, match, and pathlib

Type hints

def process(prompt: str, tokens: int = 1024) -> str:
    """Type hints make function signatures self-documenting."""
    return prompt[:tokens]

Dataclasses — clean data containers

from dataclasses import dataclass, field

@dataclass
class APIConfig:
    model: str = "gpt-4o-mini"
    temperature: float = 0.7
    max_tokens: int = 1024
    stream: bool = False

config = APIConfig(temperature=0.3)
print(config.model)       # gpt-4o-mini

pathlib — modern file paths

from pathlib import Path

data_dir = Path("data")
csv_file = data_dir / "sales.csv"

print(csv_file.exists())       # True / False
print(csv_file.stem)          # "sales"
print(csv_file.suffix)        # ".csv"
for f in data_dir.glob("*.csv"):
    print(f)

Use dataclasses for any object that is mainly data — API configs, database rows, model results. They give you __repr__, __eq__, and type hints for free.

Why uv?

The modern replacement for pip + venv + pyenv

The old problem

Traditional Python dependency management requires three separate tools: pyenv to manage Python versions, venv to create virtual environments, and pip to install packages. They work but they are slow, inconsistent, and add friction to every project setup.

uv solves all three

10–100× faster
Written in Rust — installs packages in seconds, not minutes
Lockfiles
Reproducible environments — same packages, every machine, every time
Python versions
Manages Python versions too — replaces pyenv
pyproject.toml
Modern project format — one file for everything

# macOS / Linux
curl -LsSf https://astral.sh/uv/install.sh | sh

powershell -c "irm https://astral.sh/uv/install.ps1 | iex"

uv is built by Astral (the same team behind Ruff). It is rapidly becoming the standard tool for Python project management. Starting new projects with uv is now the recommended approach.

Using uv

Create, manage, and run Python projects with uv

# Create a new project (generates pyproject.toml, .gitignore, etc.)
uv init my-ai-project
cd my-ai-project

# Add a dependency (installs + records in pyproject.toml)
uv add openai anthropic pandas

# Remove a dependency
uv remove requests

# Run a script
uv run main.py

# Sync environment from pyproject.toml (after cloning a project)
uv sync

The pyproject.toml file

# pyproject.toml — replaces requirements.txt
[project]
name = "my-ai-project"
version = "0.1.0"
requires-python = ">=3.11"
dependencies = [
    "openai>=1.0.0",
    "anthropic>=0.20.0",
    "pandas>=2.0.0",
]

When using uv, replace pip install X with uv add X, and python script.py with uv run script.py. uv manages the virtual environment automatically.
